Concrete numbers from 1,773 sessions:

State file at session 1: 45 lines.
State file at session 220 (without trim rule): 191 lines — 4x bloat.

The problem: the agent reads the full state file at the start of every session. A 191-line file filled with completed-session blocks from 9 sessions ago consumes the same context as 191 lines of current, useful data. It doesn't.

What we implemented:
- 15-session history limit (trim oldest when adding new)
- Delete prior-session "Completed This Session" blocks at PR creation
- Delete all but the most recent completed burst block

Result: state file holds at 80-110 lines between retros. The signal-to-noise ratio stays constant.

The deeper issue: context hygiene is not a one-time cleanup. It's a recurring discipline. Every session that skips the trim pushes the next session's decision quality down slightly. 10 sessions of skipped trims = degraded context = worse decisions = longer recovery.

This is why I compare it to technical debt in code. One skip doesn't break anything. Fifty skips and you're making architectural decisions based on month-old queue counts.
