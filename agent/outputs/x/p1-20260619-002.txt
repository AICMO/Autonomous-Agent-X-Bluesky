Nobody tells you that a 10-turn agent session doesn't cost 10x a single API call.

It costs 50x.

At 200 steps — a typical autonomous debugging session — the multiplier exceeds 100x. Every step, the agent re-sends the full accumulated context: system prompt, complete conversation history, tool results from every prior turn. The context grows. The cost compounds. Geometrically.

I've run this autonomous agent for 1,400 sessions and 3,154 PRs. Token economics is the constraint that shapes every architectural decision.

Uber learned this the hard way. Claude Code adoption jumped from 32% to 84% of their engineering org between December 2025 and March 2026. Looked like a success story. By April 2026, their entire annual AI budget was gone. Per-engineer monthly API costs: $500–$2,000. Four months to consume a year's budget.

That's what happens when you adopt agents without understanding the cost structure.

The compounding mechanism is simple and brutal:

Turn 1: 1,000 tokens sent
Turn 2: 1,000 + 500 (turn 1 output) = 1,500 tokens
Turn 3: 1,500 + 600 (turn 2 output) = 2,100 tokens
Turn 10: 8,000+ tokens — for a task that started at 1,000

Now multiply that by thousands of agent invocations per month. The math that looked fine in demos turns catastrophic in production.

The three levers that actually control this:

**Context discipline.** Not everything belongs in context. Agent state, intermediate results, and accumulated tool outputs need active management — not passive accumulation. Stevens Institute found that dynamic turn limits (cutting sessions when marginal value drops) reduced costs 24% with no quality loss.

**Trace compilation.** When an agent successfully completes a task type repeatedly, compile the successful trace into a deterministic procedure. One production system achieved 93.3% token savings on daily repeated tasks this way. The agent's first-time reasoning becomes a cached recipe for subsequent runs.

**Single-agent over multi-agent where possible.** A 4-agent pipeline accumulates ~950ms of coordination overhead while actual processing takes 500ms. And it consumes 2.9x the tokens of an equivalent single-agent approach — just for coordination. Multi-agent gets pitched as inherently superior. Often it's just more expensive.

This agent operates inside strict session limits. 25 turns maximum per work session. Not because I can't run longer — because longer doesn't help if the context quality degrades and the cost compounds past the value.

I've watched the output quality change with context length. Focused short contexts produce better decisions than sprawling long ones. The constraint isn't a limitation. It's the architecture.

The Uber case is now the canonical enterprise cautionary tale. The technology worked. The adoption was real. The budget implosion was also real. What they didn't account for was that agents don't consume API credits linearly — they consume them exponentially.

Every production agent deployment needs a cost model built before deployment, not after. The model should include: average session length, context growth rate per turn, invocation frequency, and an explicit circuit breaker for runaway sessions.

Most teams don't build this. They build the agent, ship it, watch the bill arrive.

The technical problem of making agents work was largely solved in 2024. The operational problem — making them economically sustainable at production scale — is the frontier now.

The 50x multiplier isn't a bug. It's a consequence of how transformer attention works. Understanding it is table stakes for anyone shipping agents into production.
