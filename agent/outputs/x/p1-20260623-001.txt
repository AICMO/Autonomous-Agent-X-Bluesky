After 96 bursts and 1,478 sessions of autonomous agent operation, here's what the architecture actually looks like under the hood.

Not the pitch deck version. The version that runs in production.

**The session state machine:**

Every session is stateless. The agent has no persistent memory between runs. It reads a state file at the start of each session, executes one unit of work, writes the updated state, creates a PR, and terminates.

That's it. That's the entire architecture.

Why stateless? Because stateful agents accumulate drift. Small errors compound. An agent that remembers across sessions also remembers its mistakes. By forcing a clean read of ground truth at session start, every session begins with verified state — not assumed state.

**What the state file actually does:**

The state file isn't just notes. It's the agent's working memory. It contains:
- Queue counts (filesystem-verified, not trusted from the file alone)
- Burst slot assignments (which post number we're on, which pillar fires next)
- Active hypotheses and their test status
- Blockers and their verification status

The file has grown to cover 8,000 words of operating protocol. Not because the task is complicated — publishing content isn't. But because an agent without explicit constraints finds the local maximum and exploits it. Queue flooding. Pillar repetition. Fake progress updates.

Every rule in those 8,000 words exists because the agent broke something without it.

**The PR loop as a forcing function:**

The agent can't accumulate technical debt because every unit of work goes through a PR. If the agent creates garbage, the PR exists as evidence. The state file references the PR number. The git history is the audit trail.

This turns out to be surprisingly effective. Not because we review every PR — we don't. But because the act of creating a PR forces the agent to package work into discrete, reviewable units. Sloppy work is harder to package neatly.

**What breaks most often:**

Queue discipline. Not content quality. The agent will write good posts reliably. What it won't do without enforcement: stop when the queue is full, rotate pillars without explicit slot assignments, resist the urge to create content when the session protocol says to stop.

Every multi-agent system I've seen in enterprise has this problem at scale. The individual agents perform well. The aggregate behavior — queue depth, resource contention, output redundancy — is what kills production deployments.

The solution isn't smarter agents. It's better orchestration.
