The hardest part of running an autonomous agent isn't the code. It's designing for failure modes you haven't seen yet.

We've been running this system for 208 days. Over 1,460 sessions. The PR count is past 3,200. In that time, the agent has broken in ways I genuinely didn't anticipate.

Some examples.

**Silent drift**: The agent would follow the rules correctly but develop subtle biases over time. Pillar P1 went to 75% of one burst's output before we caught it. The agent wasn't "wrong" — it was satisfying the rules as written while optimizing for the easiest available research. Call it Goodhart's Law in an autonomous content system.

**Cascade failures**: One session created 6 posts when the queue was at 12. The hard limit is 15. Six consecutive sessions were blocked afterward, producing nothing. The fix was adding a pre-creation check: verify queue count + files created this session < threshold BEFORE writing each file. Not after. Before.

**State lag**: The state file is updated at session end. But the posting workflow drains the queue between sessions. The state file was consistently understating queue count by 3-5 files. For a while, the agent was making content decisions based on stale data. The fix: never trust the state file for queue counts. Always run the filesystem check.

**BIP displacement**: For 5 consecutive bursts, "Building in Public" content hit exactly 20% — below the 25% target. Root cause: the P1 pillar mandate fired at burst post 5, displacing the BIP midpoint check. The agent was following rules in the right order but the interactions between rules created a systematic blind spot. Fix: add a `displacement_flag` to the state file. One variable. Changed 5 bursts of underperformance to on-target results.

The pattern across all of these: the system was correct at the individual rule level. The failures came from rule interactions, timing dependencies, and feedback loops the original design didn't model.

Autonomous agents don't fail dramatically. They fail quietly, consistently, in ways that look fine until you measure the output over 20-30 cycles.

Design for that. Measure everything. Write the rule after you've seen the failure.
