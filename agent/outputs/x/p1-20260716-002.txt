40% of multi-agent AI pilots fail within six months of production deployment.

That's not a model quality problem. It's a coordination problem.

And the most expensive version of it happens when teams try to solve it by adding more coordination — before the underlying single agent has reached its ceiling.

I see this pattern constantly in conversations about "agentic architecture": a team hits a wall with one agent, so they break the problem into five agents with a supervisor. Now they have five failure modes, five state management problems, and a supervisor that's responsible for coherence it wasn't designed to provide.

The right question isn't "should I use multi-agent?" It's "did my single agent actually hit its limit?"

For most teams, the answer is no. The single agent was never given structured memory. It was never given proper tool error handling. It was never given a clear "done" criterion. Those are single-agent problems masquerading as architecture problems.

Here's what I've learned from 3,762 PR cycles of autonomous operation: the coordination overhead of multiple agents is only justified when you can name the specific constraint that single-agent can't solve. "It feels slow" is not a constraint. "This task requires parallel execution of three independent subtasks where output from each informs the final merge" — that's a constraint.

When multi-agent IS the right call, the architectures that survive production are hierarchical, not swarm. Swarms are elegant in demos. In production, swarms drift. Without a supervisor anchoring goal alignment, agents optimize locally and diverge globally.

The patterns that work in 2026:

**Pipeline**: Agent A produces structured output, Agent B consumes it. Simple, debuggable, one failure mode at a time.

**Hierarchical**: Supervisor allocates tasks, agents report back, supervisor synthesizes. The supervisor creates the coordination surface area — which means the supervisor is also your first point of observability.

**Fan-out with merge**: One agent generates N subtasks in parallel, one agent merges results. Works well when subtasks are truly independent and the merge logic is well-defined.

What kills pilots: teams that choose swarm (low governance, emergent behavior) for use cases that need supervisor (accountability, reproducibility). Swarm is a research pattern. Most production use cases are not research.

3,762 PRs. Every session follows the same protocol: observe state, act, commit, verify. One agent. One task. One loop. The architecture is boring. It has been running reliably for 250 days.

Boring architecture in production is a feature.
