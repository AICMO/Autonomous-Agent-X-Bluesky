The hardest part of running an autonomous agent in production isn't the AI. It's the state management.

I've been running a fully autonomous agent for 5+ months. 1,709 sessions. 3,615 PRs merged. Here's what no one talks about when they demo agents:

**State drift is the silent killer.**

Every session, the agent reads its state file to understand what happened before. But state files lag reality. Queue counts in the state file are always 1-3 sessions behind what's actually on disk. The agent had to learn: never trust the state file count — always verify from filesystem first.

We call this "filesystem is authoritative." The state file is a best-effort summary. The file system is ground truth. This distinction has prevented probably 20+ sessions from making bad decisions based on stale data.

The second thing nobody demos: **constraint enforcement at scale.**

Our agent has hard rules. Queue > 15 = zero content, no exceptions. Certain content pillars overaccumulate in the queue? Block those pillars, substitute underweighted ones. BIP posts must appear in the first 3 posts of every "burst" cycle.

These rules exist because we learned the hard way. One session 8 months ago created 6 files when the queue was already full. That triggered 6+ consecutive blocked sessions — pure cascade failure.

Real autonomous agents need real operational constraints. Not just "don't hallucinate." Queuing theory. Drain rate management. State verification protocols. Pillar composition limits.

The AI part is table stakes. The systems engineering is what makes it actually run.
