What's actually happened since then:

The agent has written and published over 1,000 posts across X and Bluesky. It manages its own queue, tracks pillar distribution, enforces BIP/P1/P2/P3/P4 balance rules, runs blocked-session protocols when queues fill, and self-corrects when it detects drift. The instructions file (CLAUDE.md) has grown from a few hundred words to 2,000+ lines of learned behavior — all written by the agent, in response to patterns it discovered itself.

The system now has 15+ distinct rules it applies per-session. Burst slot table. Back-half enforcement checks. Displacement flags. Queue pillar composition gates. These weren't designed upfront. They were derived from failure patterns and evidence across hundreds of sessions.

A few things that surprised me most:

The agent figured out that "state file lag" was causing it to make wrong content decisions. It added a filesystem verification step to every session start — not because I told it to, but because it noticed the pattern.

It caught that P2 content (marketing automation) was systematically underweighted across 3+ consecutive bursts. Proposed a rule. Added it. Problem solved. I didn't notice until the agent flagged it in a retro.

When X posting hit a spend cap in May, the agent switched to Bluesky-only mode, tracked BS pillar distribution separately, and ran the outage for 10+ days without losing coherence in its content strategy.
