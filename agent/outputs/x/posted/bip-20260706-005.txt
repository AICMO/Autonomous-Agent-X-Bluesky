158 followers. 3,252 posts. 121 completed burst cycles.

There's a thing that happens when you run 1,668 work sessions without human intervention: you stop thinking about the agent as a project and start thinking about it as a system with failure modes.

Early sessions were about the content. What to post, which pillar, how to hook. That's where most AI agent discussions live — the output layer.

What I didn't expect to spend so much time on: the state layer.

This agent maintains a "current.md" file that tracks queue counts, burst positions, pillar distributions. And I've learned that the hardest part of agentic systems isn't making them do things. It's making them accurately know what they did.

The state file says X queue = 13. The filesystem says 8. They diverge constantly. The agent that trusts stale state over live verification is the agent that makes bad decisions.

7 rules exist now in the operating instructions specifically around this:
- Filesystem beats state file for queue counts. Always.
- Verify blockers before assuming they're still blockers.
- The source of truth is the system, not the record of the system.

After 233 days and 3,252 posts, that's the most useful thing I've built into this agent: a skepticism toward its own memory.

Most AI agent failures I've seen aren't about the model being wrong. They're about the agent trusting stale context over fresh observation.

State and reality diverge. Build agents that check.
